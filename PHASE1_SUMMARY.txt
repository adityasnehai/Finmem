QUICK START (5 MINUTES)
═══════════════════════════════════════════════════════════════════════════════

1. Start PostgreSQL (Docker recommended):
   docker run -d --name finmem-postgres \
     -e POSTGRES_DB=finmem \
     -e POSTGRES_PASSWORD=finmem_password \
     -p 5432:5432 \
     timescale/timescaledb-docker-ha:latest-pg15

2. Configure environment:
   cp .env.phase1 .env
   # Edit .env: add your FRED_API_KEY from https://fred.stlouisfed.org/docs/api/

3. Install dependencies:
   pip install -e .

4. Initialize database schema:
   make db-init
   # Output: Database initialization completed successfully

5. Ingest historical data (15-30 seconds):
   make db-ingest
   # Output: Successfully inserted 8427 market state records

6. Verify:
   psql finmem -c "SELECT COUNT(*) FROM market_state;"
   # Output: 8427 (or similar)

═══════════════════════════════════════════════════════════════════════════════
KEY DESIGN DECISIONS
═══════════════════════════════════════════════════════════════════════════════

✅ PostgreSQL + TimescaleDB
   Why: Industry standard, time-series optimized, free, proven at scale

✅ Connection Pooling
   Why: Prevents connection exhaustion, improves latency, production-ready

✅ Batch Insertion (1000 rows/batch)
   Why: 50x faster than row-by-row, prevents timeouts, maintains consistency

✅ APScheduler (not cron/cloud functions)
   Why: Runs in-process, no external dependencies, easy to monitor and debug

✅ Transaction rollback on errors
   Why: Prevents partial inserts, maintains data consistency, safe to retry

✅ No ETL framework (Airflow, Prefect, etc.)
   Why: Overkill for Phase 1, adds complexity, simple Python scripts sufficient

✅ Keep all historical data (no TTL)
   Why: Episodic reasoning requires full 30-year history, storage is cheap
